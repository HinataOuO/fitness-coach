# R2.1 Migrare struttura profili

## Title
Migrare struttura profili R2.

## Goal
Adottare una radice unica `Profiles/` con profilo, piano, artefatti e history isolati per atleta.

## Layer
backend

## Skill
domain-feature

## Allowed files
- `references/profile-schema.md`
- `.gitignore`
- `Profiles/<Nome>/profile.md`
- `Profiles/<Nome>/plan.md`
- `Profiles/<Nome>/artifacts/**`
- `Profiles/<Nome>/history/**`
- Directory legacy atleta top-level contenenti `profile-core.md`, `profile-plan-current.md`, `profile-log-history.md` o `profile-plans-archive.md`, esclusivamente durante migrazione manuale autorizzata.

## Out of scope
- Modifiche a dispatcher o skill fitness.
- Migrazione automatica o fallback permanente verso directory/file legacy.
- Modifiche ai contenuti tecnici dei piani o report durante spostamento.
- Commit di dati personali o artefatti atleta.

## Context to load
- `.ai-project/runtime/skills/domain-feature/SKILL.md`
- `.ai-project/runtime/skills/ponytail/SKILL.md`
- `.ai-project/local/project/PROJECT_INDEX.md`
- `references/profile-schema.md`
- `.gitignore`
- R2.0 completato e relativo mapping.
- Per ogni atleta migrato, solo i quattro file legacy di quell'atleta; mai più profili insieme.

## Implementation instructions
- Definire in `references/profile-schema.md` questa struttura canonica:

```text
Profiles/<Nome>/
├── profile.md
├── plan.md
├── artifacts/
└── history/
    ├── last-week.md
    ├── weeks/
    │   └── W<N>-<YYYY-MM-DD>.md
    └── plans/
        └── <YYYY-MM-DD>-<3|6|9|12>m.md
```

- `profile.md` conserva dati atleta, benchmark correnti, `Last log`, `Plan week`, durata ciclo e date inizio/fine ciclo.
- `plan.md` contiene solo piano madre del ciclo corrente.
- `history/last-week.md` contiene ultimo report settimanale completo; prima di sostituirlo, archiviarlo in `history/weeks/` con nome stabile e non sovrascrivibile.
- `history/plans/` contiene un file Markdown separato per ogni ciclo concluso; mai un archivio monolitico.
- `artifacts/` contiene JSON e HTML generati per quell'atleta.
- Aggiornare `.gitignore` per ignorare l'intera `/Profiles/` con motivazione privacy. Rimuovere regole nominali legacy solo dopo migrazione manuale completata o conferma che quei profili non esistono.
- Documentare procedura manuale: selezionare un atleta; creare struttura; mappare quattro file legacy secondo R2.0; dividere log e piani senza cambiare testo tecnico; verificare conteggi/date; rimuovere origine solo dopo confronto; ripetere per atleta successivo.
- Se dati legacy sono ambigui, fermare migrazione di quell'atleta e chiedere conferma. Non inventare date, settimane o ciclo.
- Dopo migrazione non introdurre letture, alias, symlink o fallback verso vecchie directory.

## Verification
- Validare struttura e campi descritti in `references/profile-schema.md`.
- Eseguire `git check-ignore -v Profiles/Example/profile.md Profiles/Example/artifacts/week-W1.json` e confermare entrambi ignorati.
- Per ogni migrazione autorizzata, confrontare contenuto e conteggio di settimane/piani prima e dopo.
- Eseguire `rg -n 'profile-core|profile-plan-current|profile-log-history|profile-plans-archive' references/profile-schema.md .gitignore` e accettare solo riferimenti nella procedura legacy esplicita.

## Done criteria
- Schema R2 definisce tutti i file, campi e nomi history.
- History settimanale e archivio piani sono per-file.
- Profili e artefatti sotto `/Profiles/` sono ignorati.
- Migrazione legacy è manuale, verificabile e senza fallback permanente.
- Nessun file fuori da `Allowed files` modificato.
