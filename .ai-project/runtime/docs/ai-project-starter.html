AI Project Starter installa regole condivise, skill operative, slash command, template memoria, roadmap e issue template. Separa package aggiornabile da stato locale del progetto.

Overview

Questo repository egrave; la sorgente template. Dopo installazione, il package vive in .ai-project/runtime/. Lo stato locale del progetto vive in .ai-project/local/ e non viene sovrascritto dagli update.

Runtime

Contiene core rules, skill, command wrapper e template gestiti dal package. Non va modificato a mano nel progetto installato.

Local state

Contiene memory shard, roadmap, overlay e fatti specifici del progetto. Va committato insieme al progetto.

Discovery copy

Sincronizza AGENTS.md, CLAUDE.md, .agents/skills/ e .claude/commands/.

Architettura

Percorso sorgente Ruolo Installazione
core/ Regole condivise: kernel, load order, convenzioni. .ai-project/runtime/core/
skills/ Skill operative portabili, una directory per skill. .ai-project/runtime/skills/
commands/claude/ Slash command Claude sottili che caricano skill. .ai-project/runtime/commands/claude/

overlays/example/ Forma esempio per overlay specifici progetto. .ai-project/local/project/overlays/example/
github/ISSUE_TEMPLATE/ Template issue scope/layer. Disponibile nel runtime; copia manuale quando serve.

starter.json Manifest: version, root, skill, command, tag, layer. Fonte runtime e validazione.

starter.json definisce canonicalRoot .ai-project, runtimeRoot .ai-project/runtime, localRoot .ai-project/local, 16 skill, 16 command wrapper, memory tag e roadmap layer.

Installazione

Install veloce nel repository corrente

curl -fsSL https://raw.githubusercontent.com/HinataOuO/AI-ProjectManager/main/scripts/ai-project.mjs | node - install-here

Install da clone locale o sorgente Git

node /path/to/AI-ProjectStarter/scripts/ai-project.mjs install <git-url-or-local-path> --version v2.0.0

Commit dopo install

git add AGENTS.md CLAUDE.md .agents .claude .ai-project.lock.json .ai-project/local
git commit -m "chore: install AI project manager"
Regola importante: committare .ai-project/local/. Contiene memoria, overlay e roadmap. Non editare a mano .ai-project/runtime/: viene gestito dal package.

CLI

Comando Uso Note
install Installa in --project <path> o cwd. Accetta Git URL o path locale; legge manifest sorgente.
install-here Installa nel repository corrente. Usa --source, env AI_PROJECT_MANAGER_SOURCE o default GitHub.
update Aggiorna runtime in cwd o --project <path>. Controlla runtime drift e non sovrascrive mai .ai-project/local/.
update-here Alias compatibile per aggiornare il repository corrente. Preferire update.
status Mostra sorgente, ref, versione, commit, drift. Legge .ai-project.lock.json.
status-here Alias compatibile per status del repository corrente. Preferire status.
sync-discovery Rigenera entrypoint e copie discovery. Scrive AGENTS.md, CLAUDE.md, .agents/skills/, .claude/commands/.

Opzioni

  • --version <ref>: branch, tag o ref Git per clone/update.
  • --project <path>: target repository; default cwd.
  • --source <git-url-or-local-path>: sorgente per install-here.
  • --force: permette overwrite di file esistenti o runtime con drift.

Comandi rapidi

node .ai-project/runtime/scripts/ai-project.mjs update
node .ai-project/runtime/scripts/ai-project.mjs status
node .ai-project/runtime/scripts/ai-project.mjs sync-discovery
bash scripts/validate-starter.sh

Aggiornare progetto installato e agenti

update scarica/sostituisce .ai-project/runtime/ dalla source configurata e aggiorna .ai-project.lock.json. Non sovrascrive .ai-project/local/. 

node .ai-project/runtime/scripts/ai-project.mjs update --version main
node .ai-project/runtime/scripts/ai-project.mjs sync-discovery --force

sync-discovery non scarica aggiornamenti: rigenera AGENTS.md, CLAUDE.md, .agents/skills/ e .claude/commands/ dal runtime locale corrente. Se non cambia nulla, eseguire prima update --version main o verificare source/ref con status. 

node .ai-project/runtime/scripts/ai-project.mjs status

Logica interna

Flusso installazione

1. Source Git URL o path locale viene materializzato. Se Git, clone temp.
2. Manifest starter.json valida root canonico e liste.
3. Runtime Package copiato in .ai-project/runtime/.
4. Local project/ e overlay esempio vengono seedati solo se mancanti.
5. Discovery Entrypoint e skill/command discovery vengono sincronizzati.
6. Lock .ai-project.lock.json salva source, ref, commit, hash runtime.
7. Commit User committa entrypoint, lock e local state.
8. Update Update successivo sovrascrive runtime dopo drift check.

Flusso task agente

  1. Carica .ai-project/runtime/core/KERNEL.md.
  2. Carica .ai-project/runtime/core/CONVENTIONS.md.
  3. Carica .ai-project/local/project/PROJECT_INDEX.md.
  4. Seleziona skill o wrapper piu piccolo per richiesta.
  5. Legge MEMORY_INDEX.md, poi solo shard con tag rilevanti.
  6. Per roadmap legge index, un MACRO.md, poi un solo layer.
  7. Legge overlay solo quando servono path, comandi, provider DB o regole dominio.
  8. Lavora scoped, verifica, propone memory delta solo se utile e confermato.

Skill

Ogni skill ha frontmatter con name e description, poi solo questi heading: purpose, load, scope, deny, procedure, done.

Skill Quando usarla Load chiave Comando
backend Server code, DB query, API route, auth, validation. Memory index, shard backend, overlay per path/auth. /backend
close-task Chiudere roadmap layer dopo conferma esplicita. Roadmap index, macro feature, layer target. /close-task
commands Elencare skill e instradare richiesta. Core, load order, project index. /commands
domain-feature Caricare contesto di una feature dominio limitata. Overlay domain, shard domain, skill layer. /domain-feature
frontend Page, component, style, client behavior, i18n UI. Memory index, shard frontend, overlay UI/i18n. /frontend
git-issue Analizzare issue e produrre piano implementazione. Issue body/comment, project index, skill e memoria rilevanti. /git-issue
i18n Solo modifiche file messaggi/traduzioni. Overlay per locale supportati e path messaggi. /i18n
log-issue Creare issue e proporre memory shard. Memory index, shard scope, overlay label/repo. /log-issue
migration Workflow DB completo: schema, migration, seed/import. Shard migration, overlay provider/script/comandi. /migration
next-task Trovare prossimo task roadmap azionabile. Roadmap index, macro corrente, un layer. /next-task
project-discovery Scoprire una area progetto con basso uso token. Project index, memory index, overlay, shard discovery/area. /project-discovery
push Commit e push dopo diff review e conferma. Solo git status, diff e branch. /push
refactor Analizzare file, funzione, modulo o topic e produrre piano refactor/ottimizzazione read-only. Memory index, shard always, shard scope rilevanti. /refactor
schema Solo cambio modello dati/schema. Shard migration, overlay DB/schema path. /schema
session-status Mostrare task list della sessione agente. Stato sessione corrente. /session-status
task-status Overview roadmap read-only. Roadmap index e macro feature wip. /task-status

Comandi Claude

I file in commands/claude/ sono wrapper sottili. Ogni wrapper carica .ai-project/runtime/skills/<name>/SKILL.md e segue load/scope/deny/procedure/done. 

# /frontend
Load `.ai-project/runtime/skills/frontend/SKILL.md`. Follow its `load/scope/deny/procedure/done`.

Le copie installate vivono in .claude/commands/ e vengono rigenerate da sync-discovery. Alias specifici progetto vanno creati fuori dal template, dopo installazione.

Memoria

La memoria locale vive in .ai-project/local/project/memory/. L'agente legge prima MEMORY_INDEX.md, poi solo shard con tag richiesti.

always backend frontend migration discovery domain roadmap

Frontmatter richiesto

---
id: short-id
tags: [always]
load: always
updated: YYYY-MM-DD
depends: []
---
  • always: preferenze utente e regole workflow durevoli.
  • backend: server, DB access, API, auth, validation.
  • frontend: UI, componenti, stili, messaggi i18n.
  • migration: schema, migration, seed/import.
  • discovery: mappa progetto, path, comandi, architettura.
  • domain: regole dominio progetto.
  • roadmap: stato/pianificazione durevole.
Nessuna scrittura automatica: ogni memory write richiede conferma esplicita. L'agente deve mostrare target, contenuto o riassunto preciso, e motivo.

Roadmap

La roadmap locale vive in .ai-project/local/project/roadmap/. Il modello usa feature macro e layer tecnici.

Elemento Ruolo
INDEX.md Lista feature correnti, stato e layer.
MACRO.md Obiettivo feature, status e checklist per layer.
database.md Schema, migration path, seed/import, recovery.
backend.md Entrypoint server/API, validation, auth/RBAC, error behavior.
frontend.md Screen/component, loading/empty/error, i18n, responsive.
verify.md Lint, typecheck, test, manual flow, gap noti.

Status

Stati supportati: todo, wip, blocked, done. Regola default: una sola feature wip, salvo overlay diverso.

Workflow

  • next-task: legge index, prima feature wip o todo, macro, poi un layer.
  • task-status: overview read-only senza layer detail.
  • close-task: chiude layer solo dopo conferma esplicita.

Sicurezza

Operazioni distruttive, irreversibili, git push, roadmap close e memory write richiedono conferma esplicita prima della scrittura.
  • task-status e session-status restano read-only.
  • close-task non scrive roadmap senza conferma.
  • push ispeziona status/diff/branch, blocca .env*, *.key, *.pem, non fa force push.
  • Operazioni DB distruttive richiedono conferma e note recovery quando necessarie.
  • Fatti specifici progetto stanno in overlay o memory, non in core/skill/command generici.
  • Modifiche utili in runtime installato vanno riportate nel source repo o in local state, non lasciate come drift.

Manutenzione

Policy contenuti

  • Prompt interni, skill e command in inglese.
  • Output utente in italiano salvo richiesta diversa.
  • File piccoli, deduplicati, caricati lazy.
  • Skill con heading fissi e nessuna sezione extra.
  • Nessun nome progetto reale, alias privato o fatto dominio nei file generici.

Runtime drift

update confronta hash runtime salvati nel lock con lo stato corrente. Se trova drift, fallisce e mostra i file. Usare --force solo per scartare modifiche runtime locali.

Validazione

Validazione principale dal repository sorgente:

bash scripts/validate-starter.sh

Oppure da parent directory:

bash AI-ProjectStarter/scripts/validate-starter.sh

Cosa controlla

  • starter.json: root, liste stringa, duplicati, file richiesti.
  • Skill: frontmatter, name coerente, description, heading obbligati.
  • Claude wrapper: un solo target skill runtime corretto.
  • Memoria: frontmatter, tag validi, depends inline list.
  • Adapter: riferimenti a .ai-project/runtime/ e .ai-project/local/.
  • Hardcode scan: blocca termini specifici progetto nei file generici.
  • Word count: monitora budget token complessivo.
